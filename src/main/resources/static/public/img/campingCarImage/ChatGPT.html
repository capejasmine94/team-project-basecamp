1. Service Interface 작성

2. Service Implementation 작성

java
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;
import java.util.Map;
import java.util.List;

@Service
public class ReservationServiceImpl implements ReservationService {

    private final ReservationMapper reservationMapper;

    @Autowired
    public ReservationServiceImpl(ReservationMapper reservationMapper) {
        this.reservationMapper = reservationMapper;
    }

    @Override
    public List<Map<String, Object>> findReservationByRentUserId(Long rentUserId) {

3. Mapper Interface 작성

이미 Mapper XML에서 select 문을 작성했으므로, 이를 호출할 수 있는 Mapper 인터페이스를 추가합니다.

java
import org.apache.ibatis.annotations.Select;
import java.util.Map;
import java.util.List;

public interface ReservationMapper {
    
    @Select("SELECT * FROM bc_reservation br " +
            "INNER JOIN bc_rent_user bru ON br.rent_user_id = bru.id " +
            "INNER JOIN bc_campingcar bc ON br.product_id = bc.id " +
            "WHERE br.rent_user_id = #{id}")
    List<Map<String, Object>> findReservationByRentUserId(Long rentUserId);
}

4. Controller 작성 (선택사항)

Service를 이용하여 데이터를 조회하는 Controller를 작성할 수도 있습니다.

java
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;
import java.util.Map;
import java.util.List;

@RestController
public class ReservationController {

    private final ReservationService reservationService;

    @Autowired
    public ReservationController(ReservationService reservationService) {
        this.reservationService = reservationService;
    }

    @GetMapping("/reservations")
    public List<Map<String, Object>> getReservationsByUserId(@RequestParam("userId") Long userId) {
        return reservationService.findReservationByRentUserId(userId);
    }
}

이렇게 하면 findReservationByRentUserId 메서드를 통해 예약 정보를 조회할 수 있게 됩니다. 이 코드는 MyBatis를 사용한 데이터베이스 접근을 가정하며, Spring Framework의 기본적인 구조를 따르고 있습니다.

추가적인 비즈니스 로직이 필요하다면 Service 레이어에 추가적으로 구현할 수 있습니다.

Reservation 테이블의 rent_user_id 외래키와, product_id 가 외래키로 들어가 있어서 위와 같이 xml을 작성했어. 이렇게 하는게 맞아?

XML Mapper에서 외래키를 사용하여 두 테이블을 조인하는 방식은 일반적으로 맞습니다. 하지만 몇 가지 중요한 점을 확인해야 합니다.

1. 조인의 적절성

  • bc_reservation 테이블이 rent_user_id와 product_id를 외래키로 참조하고 있기 때문에, bc_rent_user와 bc_campingcar 테이블과 조인하는 것은 올바릅니다.
  • XML Mapper에서 작성한 SQL은 이 조인을 수행하며, rent_user_id를 기준으로 예약 정보를 조회합니다.

2. 컬럼 명시

  • SELECT * 대신 필요한 컬럼만 명시하는 것이 성능 면에서 유리합니다. SELECT *는 모든 컬럼을 가져오기 때문에, 필요하지 않은 데이터를 불필요하게 가져올 수 있습니다.
  • 조인된 테이블들에 동일한 이름의 컬럼이 있다면, 이를 명시적으로 구분해야 합니다. 예를 들어, br.id AS reservation_id 와 같이 별칭을 사용하는 것이 좋습니다.

3. SQL 작성 개선

  • SQL 문을 조금 더 구체적으로 작성해보겠습니다. 필요한 컬럼을 명시적으로 지정하고, 성능과 가독성을 향상시키는 방향으로 개선합니다.

개선된 XML Mapper 예시

xml
<!-- 예약 정보 확인 및 예약관리 --> 
<select id="findReservationByRentUserId" resultType="Map">
    select 
        br.id as reservation_id,
        br.product_id,
        br.rent_user_id,
        br.start_date,
        br.end_date,
        bru.name as user_name,
        bru.email as user_email,
        bc.name as car_name,
        bc.model as car_model
    from bc_reservation br 
    inner join bc_rent_user bru on br.rent_user_id = bru.id
    inner join bc_campingcar bc on br.product_id = bc.id
    where br.rent_user_id = #{id}
</select>

주의 사항

  • 조인 방식: INNER JOIN은 조인 조건에 만족하는 레코드만 반환합니다. 만약 rent_user_id에 대한 예약이 없거나, 조인된 테이블에서 매칭되는 레코드가 없는 경우, 해당 유저에 대한 정보는 반환되지 않습니다. 조인의 방식 (LEFT JOIN 등)이 적절한지 확인하세요.
  • 외래키 제약 조건: SQL 문 외에도 데이터베이스 스키마에서 외래키 제약 조건이 올바르게 설정되어 있는지 확인해야 합니다.

결론

  • 작성하신 XML Mapper는 외래키 기반으로 조인을 수행하는 데 적합하며, 올바르게 작성된 것으로 보입니다.
  • 다만, SELECT *를 피하고 필요한 컬럼을 명시적으로 지정하는 것이 바람직합니다. 이는 성능 개선과 코드의 명확성을 높이는 데 도움이 됩니다.
외래키 테이블의 컬럼명을 일부 써야하기 때문에 *로 하고 싶었던 건데.. 너가 추천한 As를 어떻게 작성해야 할지 모르겠어.

SELECT * 대신 특정 컬럼을 선택하면서 AS를 사용하여 컬럼에 별칭을 지정하는 방법은 아래와 같습니다. 이는 컬럼 이름이 겹치는 경우에도 명확하게 데이터를 가져올 수 있게 해줍니다.

xml

컬럼이 많은 경우

만약 모든 컬럼을 명시적으로 나열하는 것이 번거롭거나, 일부 컬럼만 필요한 경우에는 다음과 같이 주요 컬럼에만 AS를 사용하고, 나머지는 그대로 SELECT *를 사용하는 것도 가능합니다.

xml
<!-- 예약 정보 확인 및 예약관리 --> 
<select id="findReservationByRentUserId" resultType="Map">
    select 
        br.*, 
        bru.name as user_name, 
        bru.email as user_email, 
        bc.name as car_name, 
        bc.model as car_model
    from bc_reservation br 
    inner join bc_rent_user bru on br.rent_user_id = bru.id
    inner join bc_campingcar bc on br.product_id = bc.id
    where br.rent_user_id = #{id}
</select>

이 방법의 장점

  • br.*: bc_reservation 테이블의 모든 컬럼을 가져옵니다.
  • bru.name as user_name: bc_rent_user 테이블의 name 컬럼을 명시적으로 가져와서, 만약 다른 테이블에서도 name 컬럼이 있다면 충돌을 피할 수 있습니다.

이렇게 하면 외래키 테이블의 주요 컬럼을 명확하게 참조하면서도, 불필요한 충돌을 방지할 수 있습니다. 필요한 컬럼에만 AS를 사용하고 나머지는 *로 처리하는 방식은 편리하면서도 코드의 가독성을 유지할 수 있는 방법입니다.

SQL Error [1064] [42000]: (conn=488789) You have an error in your SQL syntax; check the manual that corresponds to your MariaDB server version for the right syntax to use near '.product_name, bc.rental_fee_weekdays, bc.rental_fee_weekend, 오류 해석해줘

SQL Error [1064] [42000] 오류는 SQL 구문에 문법적 오류가 있을 때 발생하는 일반적인 오류 메시지입니다. 이 오류 메시지에서 중요한 부분은 near '.product_name, bc.rental_fee_weekdays, bc.rental_fee_weekend입니다. 여기서 . 앞에 오는 내용이 누락되었거나, 잘못된 구문이 사용된 경우가 원인일 수 있습니다.

문제점 분석

  1. 잘못된 테이블 또는 별칭 사용:

    • . 앞에 테이블 이름이나 테이블의 별칭이 와야 하는데, 이 부분이 잘못되었거나 누락된 것일 수 있습니다. 예를 들어, bc. 앞에 잘못된 테이블 이름이나 별칭이 사용되었을 수 있습니다.

  2. 필드명 오타:

    • .product_name 부분에서 . 앞에 올바른 테이블 별칭이 없으면 오류가 발생합니다. 예를 들어 bc.product_name처럼 테이블 별칭을 붙여야 합니다.

  3. SQL 구문이 잘못된 경우:

    • 일반적으로 SELECT 문에서 특정 테이블의 컬럼을 지정할 때, 테이블명.컬럼명 또는 테이블별칭.컬럼명의 형태로 사용합니다.

해결 방법

아래와 같은 방법으로 SQL 구문을 수정해 볼 수 있습니다.

  1. 테이블 또는 별칭이 잘못되었을 경우:

    sql
    select

    bc.rental_fee_weekdays,
    bc.rental_fee_weekend
from bc_campingcar bc
    • 각 컬럼 앞에 올바른 테이블 이름 또는 별칭을 붙입니다.

  2. 필드명 오타 수정:

    • 필드명이 정확한지, . 앞에 올바른 테이블 이름 또는 별칭이 있는지 확인합니다. 예를 들어 bc.product_name 대신 product_name만 사용하면 오류가 발생할 수 있습니다.

  3. 전체 SQL 문법 다시 확인:

    • SQL 구문 전체를 다시 확인하여 문법적으로 올바른지 확인합니다. 필요한 경우 SQL 문 전체를 공유해 주시면 더 구체적인 오류를 찾고 해결할 수 있습니다.

그러면 resultType=Map을 지정하면 interface와 service는 어떻게 작성하면 되??

resultType=Map을 지정한 경우, MyBatis는 쿼리 결과를 Map<String, Object> 형식으로 반환합니다. 이때, 키는 SQL 쿼리에서 선택된 컬럼의 이름(별칭 포함)이며, 값은 해당 컬럼의 값이 됩니다.

1. Mapper Interface 작성

resultType=Map을 사용할 때, MyBatis Mapper Interface는 List<Map<String, Object>>를 반환하도록 설정합니다.

java
import java.util.List;
import java.util.Map;

public interface ReservationMapper {

    List<Map<String, Object>> findReservationByRentUserId(Long rentUserId);
}

2. Service Interface 작성

Service Interface에서도 Mapper에서 반환한 결과를 그대로 반환할 수 있도록 List<Map<String, Object>>를 사용합니다.

java
import java.util.List;
import java.util.Map;

public interface ReservationService {
    List<Map<String, Object>> getReservationByRentUserId(Long rentUserId);
}

3. Service Implementation 작성

Service Implementation에서는 Mapper를 주입받아 호출하고, 그대로 결과를 반환하면 됩니다.

java
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;
import java.util.List;
import java.util.Map;

@Service
public class ReservationServiceImpl implements ReservationService {

    private final ReservationMapper reservationMapper;

    @Autowired
    public ReservationServiceImpl(ReservationMapper reservationMapper) {
        this.reservationMapper = reservationMapper;
    }

    @Override
    public List<Map<String, Object>> getReservationByRentUserId(Long rentUserId) {
        return reservationMapper.findReservationByRentUserId(rentUserId);
    }
}

4. Controller 작성 (선택사항)

Controller를 통해 Service를 호출하고, 결과를 반환하도록 할 수 있습니다.

java
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;
import java.util.List;
import java.util.Map;

@RestController
public class ReservationController {

    private final ReservationService reservationService;

    @Autowired
    public ReservationController(ReservationService reservationService) {
        this.reservationService = reservationService;
    }

    @GetMapping("/reservations")
    public List<Map<String, Object>> getReservationsByUserId(@RequestParam("userId") Long userId) {
        return reservationService.getReservationByRentUserId(userId);
    }
}

설명:

  1. Mapper Interface:

    • findReservationByRentUserId 메서드는 Long rentUserId를 파라미터로 받아, List<Map<String, Object>> 타입의 결과를 반환합니다.

  2. Service Interface:

    • Service 계층에서도 동일하게 List<Map<String, Object>>를 반환하도록 메서드를 정의합니다.

  3. Service Implementation:

    • Service Implementation에서는 Mapper의 메서드를 호출하여 그대로 반환합니다. 추가적인 비즈니스 로직이 필요하다면 이 부분에서 처리할 수 있습니다.

  4. Controller (선택사항):

    • 컨트롤러에서 Service 메서드를 호출하여, 사용자로부터 받은 userId를 기반으로 예약 내역을 조회할 수 있습니다.

이와 같이 설정하면 MyBatis가 쿼리 결과를 Map 형태로 반환하고, 이 결과를 서비스 계층과 컨트롤러 계층에서 쉽게 활용할 수 있습니다.

왜 List<Map<Strin,Objcet>>로 반환 값을 해야하는거야?

List<Map<String, Object>>로 반환 값을 사용하는 이유는 다음과 같습니다:

1. 쿼리 결과가 여러 행일 수 있음:

  • SQL 쿼리 결과는 다수의 행을 반환할 수 있습니다. List<Map<String, Object>>는 이 경우 각 행을 Map<String, Object> 형태로 담아 여러 행을 리스트로 관리할 수 있게 합니다. List는 쿼리 결과의 여러 행을 관리하기 위해 사용됩니다.

2. Map을 사용하는 이유:

  • Map<String, Object>는 한 행의 데이터를 키-값 쌍으로 저장합니다. 여기서 String은 컬럼 이름이나 별칭을, Object는 그 컬럼의 값을 나타냅니다.
  • SQL 쿼리에서 반환되는 컬럼의 이름을 Map의 키로 사용하고, 각 컬럼의 값을 Map의 값으로 저장하게 됩니다. 이를 통해 어떤 데이터가 어떤 컬럼에 해당하는지 쉽게 접근할 수 있습니다.

3. 동적 컬럼 처리:

  • Map<String, Object>는 컬럼 수와 이름이 동적으로 변할 수 있는 경우 유연하게 데이터를 다룰 수 있습니다. 예를 들어, 쿼리의 결과가 항상 동일한 구조가 아닐 때, 특정 클래스에 매핑하지 않고 동적으로 결과를 처리할 수 있습니다.
  • 특정 클래스에 매핑하는 것보다 덜 엄격하게 구조를 다루고 싶을 때 유용합니다.

4. 비즈니스 요구사항에 따른 유연성:

  • 비즈니스 로직에 따라 다양한 컬럼이 필요할 수 있고, 이 컬럼들이 변경될 가능성이 있습니다. 이럴 때, Map을 사용하면 새로운 컬럼이 추가되거나 삭제되어도 코드의 변경이 최소화됩니다.

예시 상황:

만약 Reservation이라는 클래스가 존재하고, 쿼리 결과가 항상 이 클래스의 필드에 정확하게 매핑된다면, List<Reservation>으로 반환할 수 있습니다. 그러나 쿼리 결과가 여러 테이블에서 조인된 데이터거나, 동적으로 선택되는 컬럼들로 이루어진다면 Map을 사용하여 데이터를 처리하는 것이 더 적합합니다.

요약:

  • **List<Map<String, Object>>**는 SQL 쿼리 결과가 여러 행으로 이루어질 때 각 행을 Map 형태로 표현하고, 이를 리스트로 묶어 반환하는 구조입니다.
  • **Map<String, Object>**는 한 행의 데이터를 컬럼 이름과 그에 대응하는 값으로 저장하는 유연한 방식입니다.
  • 이는 다양한 쿼리 결과를 손쉽게 처리하고, 동적으로 변화하는 컬럼 구조에 대응하기 위해 사용됩니다.
특정 user에 대한 것인데 List<Map<String,Object>>를 사용해도 되?

특정 사용자에 대한 데이터를 조회하는 경우에도 List<Map<String, Object>>를 사용할 수 있습니다. 그러나 그 사용이 적절한지는 쿼리 결과가 몇 개의 행을 반환할지에 따라 달라집니다.

1. 쿼리 결과가 한 행만 반환될 경우:

  • 특정 사용자에 대한 예약 내역을 조회하는 쿼리가 항상 하나의 행만 반환할 것으로 확실하다면, List<Map<String, Object>> 대신 **Map<String, Object>**를 사용하는 것이 더 적합할 수 있습니다.
  • Map<String, Object>는 한 행의 데이터를 키-값 쌍으로 표현하므로, 이 경우 리스트를 사용하지 않아도 됩니다.

예를 들어:

java
public interface ReservationMapper {
    Map<String, Object> findReservationByRentUserId(Long rentUserId);
}

public interface ReservationService {
    Map<String, Object> getReservationByRentUserId(Long rentUserId);
}

2. 쿼리 결과가 여러 행을 반환할 수 있는 경우:

  • 특정 사용자에 대한 여러 예약 내역을 조회하거나, 쿼리가 여러 행을 반환할 가능성이 있다면, List<Map<String, Object>>를 사용하는 것이 맞습니다.
  • 여러 행을 반환할 경우 각 행은 Map<String, Object>로 표현되고, 이들을 리스트로 묶어 반환합니다.

예를 들어:

java
public interface ReservationMapper {
    List<Map<String, Object>> findReservationByRentUserId(Long rentUserId);
}

    List<Map<String, Object>> getReservationByRentUserId(Long rentUserId);
}

3. 결정 기준:

  • 단일 행: 특정 사용자에 대한 결과가 단 하나의 예약 내역 또는 하나의 행만 반환된다면 Map<String, Object>를 사용합니다.
  • 여러 행: 특정 사용자에 대한 여러 예약 내역 또는 여러 행이 반환될 가능성이 있다면 List<Map<String, Object>>를 사용합니다.

결론:

  • 쿼리 결과가 항상 한 행만 반환될 것으로 확신된다면, Map<String, Object>를 사용하는 것이 더 적절합니다.
  • 만약 여러 행이 반환될 가능성이 있다면 List<Map<String, Object>>가 더 적합합니다.

따라서, 쿼리 결과가 어떻게 나올지를 기준으로 어떤 타입을 사용할지 결정하면 됩니다.

그러면 사용자가 현재 차량에 대한 예약 정보를 하나의 행만 반환 한다면 쿼리를 다시 수정해야 하는거야?

만약 특정 사용자의 특정 차량에 대한 예약 정보가 항상 하나의 행만 반환되도록 보장된다면, 쿼리를 수정할 필요는 없습니다. 대신, 결과를 처리하는 방식만 변경하면 됩니다.

1. 쿼리 수정이 필요 없는 경우

  • 쿼리는 사용자가 현재 예약한 차량에 대한 정보를 반환하고, 해당 결과가 항상 하나의 행을 반환할 것으로 확신할 수 있다면 쿼리를 그대로 사용하면 됩니다.
  • 이 경우 Mapper와 Service에서 Map<String, Object> 타입으로 반환하도록 변경하면 됩니다.

2. Mapper Interface와 Service Interface의 수정

이제 쿼리 결과를 Map<String, Object>로 받을 수 있도록 Mapper와 Service를 수정해보겠습니다.

Mapper Interface

    Map<String, Object> findReservationByRentUserId(Long rentUserId);
}

Service Interface

java
import java.util.Map;

public interface ReservationService {
    Map<String, Object> getReservationByRentUserId(Long rentUserId);
}

Service Implementation

java
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Service;
import java.util.Map;

@Service
public class ReservationServiceImpl implements ReservationService {

    private final ReservationMapper reservationMapper;

    @Autowired
    public ReservationServiceImpl(ReservationMapper reservationMapper) {
        this.reservationMapper = reservationMapper;
    }

    @Override
    public Map<String, Object> getReservationByRentUserId(Long rentUserId) {
        return reservationMapper.findReservationByRentUserId(rentUserId);
    }
}

  • 쿼리 자체는 수정할 필요가 없습니다. 쿼리에서 반환되는 결과가 하나의 행일 경우, Map<String, Object> 타입으로 결과를 받으면 됩니다.
  • 이 Map에서 키는 컬럼의 이름(혹은 별칭), 값은 해당 컬럼의 값으로 매핑됩니다.

4. 결과가 여러 행일 경우

  • 만약 쿼리가 여러 행을 반환할 가능성이 있다면, List<Map<String, Object>>를 사용해야 합니다.
  • 그러나 특정 사용자와 특정 차량에 대한 예약 정보가 하나의 행만 반환되도록 설계되었다면, Map<String, Object>로 충분합니다.

결론:

  • 쿼리를 수정할 필요는 없으며, Mapper와 Service에서 결과를 Map<String, Object>로 받도록 변경하면 됩니다.
  • 이 방법은 쿼리가 항상 하나의 행을 반환하는 경우에 적합하며, 여러 행이 반환될 가능성이 없다면 이 방법을 사용하는 것이 맞습니다.

사용자가 특정 차량에 대해 정보를 확인하고 "예약하기" 버튼을 누르면, 최종 예약 내역을 확인하기 위한 쿼리로 작성하신 쿼리는 적절해 보입니다. 특히, 사용자가 선택한 특정 예약(br.id = 13)과 특정 사용자(br.rent_user_id = 14)를 기준으로 예약 내역을 조회하는 쿼리이므로, 이 조건들이 맞다면 쿼리는 정확하게 작동할 것입니다.

다만, 최종 확인을 위해 몇 가지 중요한 점들을 다시 한번 검토해 보겠습니다:

1. 쿼리 목적의 적합성

  • 사용자가 특정 차량을 선택하고, 예약 정보를 확인하는 과정에서, WHERE 조건을 통해 rent_user_id와 reservation_id가 올바르게 필터링되고 있습니다.
  • br.id는 예약의 고유 ID이며, br.rent_user_id는 예약을 한 사용자의 ID입니다. 두 조건이 동시에 만족되어야 해당 예약에 대한 정보가 반환됩니다.

2. 조인된 테이블 관계

  • 쿼리 내에 사용된 조인들이 적절하게 설정되었습니다. 각 테이블의 관계는 외래키를 통해 잘 연결되어 있고, 필요한 정보들을 가져오는 데 문제가 없어 보입니다.
  • 조인된 테이블에서 필요한 모든 정보를 쿼리에 포함시켜, 사용자가 예약한 차량의 상세 정보(예: 차종, 대여 요금, 운전 조건 등)까지 포함한 최종 예약 내역을 조회할 수 있습니다.

3. 쿼리 결과의 기대

  • 쿼리가 특정 예약 ID와 특정 사용자 ID를 기준으로 조회되기 때문에, 결과는 하나의 행을 반환할 것으로 예상됩니다.
  • 이 행에는 사용자가 예약한 차량의 모든 관련 정보와 사용자의 정보가 함께 포함되어, 최종 예약 내역을 확인하는 데 필요한 모든 데이터를 제공합니다.

4. 쿼리의 타당성

  • 쿼리 내에서 조인된 모든 테이블이 올바르게 참조되고 있으며, 필요한 컬럼들에 대해 별칭을 사용하여 가독성을 높였습니다.
  • WHERE 조건은 사용자와 특정 예약을 정확히 필터링하도록 설계되었으므로, 이 쿼리는 목적에 맞게 작동할 것입니다.

최종 결론:

  • 작성하신 쿼리는 정확합니다. 이 쿼리는 사용자가 특정 차량에 대해 예약 정보를 확인한 후, 최종 예약 내역을 조회하는 데 적합합니다.
  • 사용자의 ID(rent_user_id)와 예약 ID(id)를 통해 정확한 예약 내역을 가져올 수 있습니다.
  • 쿼리의 논리와 조인된 테이블 관계가 잘 맞기 때문에, 쿼리 결과는 기대하는 대로 나올 것입니다.

이 쿼리를 사용하면, 사용자가 예약하려는 차량에 대한 최종 확인 정보를 정확히 조회할 수 있습니다.

3. **CTE (Common Table Expressions) 사용

쿼리 작성과 데이터베이스 작업은 처음에는 복잡하고 어려울 수 있지만, 조금씩 익숙해지면 더 쉽게 이해할 수 있습니다. 현재 하시는 작업은 특정 사용자의 특정 예약 정보를 정확하게 조회하는 것으로, 이 작업을 다른 방법으로 접근하는 방법을 간단히 설명해 드리겠습니다.

1. 뷰(View)를 사용한 접근

  • 만약 동일한 쿼리를 자주 사용해야 하거나, 복잡한 조인과 조건을 반복적으로 사용할 경우, **데이터베이스 뷰(View)**를 생성할 수 있습니다.
  • 뷰는 미리 정의된 SQL 쿼리의 결과를 테이블처럼 사용할 수 있게 해주는 기능입니다. 예를 들어, 최종 예약 정보를 반환하는 뷰를 만들어 놓고, 단순히 뷰에서 데이터를 조회할 수 있습니다.
sql
CREATE VIEW user_reservation_view AS
select 
    br.id,
    br.start_date as reservationStart_date,
    br.end_date as reservationEnd_date,
    br.progress as reservationProgress,
    br.created_at as reservationCreated_at,
    bu.nickname as rentUserNickname,
    brci.comp_name,
    bctc.name as carTypename,
    bc.product_name,
    bc.rental_fee_weekdays,
    bc.rental_fee_weekend,
    bc.garage_address, 
    bc.seating_capacity,
    bc.sleeping_capacity,
    bdlc.name AS driver_license_name,  
    bdac.name AS driver_age_condition_name, 
    bdec.name AS driver_experience_condition_name
from bc_reservation br 
inner join bc_rent_user bru on br.rent_user_id = bru.id
inner join bc_user bu  on bru.user_id = bu.id
inner join bc_campingcar bc on br.product_id = bc.id
inner join bc_rental_company_info brci on bc.rental_company_id = brci.id
inner join bc_car_type_categories bctc on bc.category_id = bctc.id
INNER join bc_driver_license_condition bdlc on bc.driver_license_id = bdlc.id
INNER join bc_driver_age_condition bdac on bc.driver_age_id = bdac.id
INNER join bc_driver_experience_condition bdec on bc.driver_experience_id = bdec.id;

이후에는 간단히 다음과 같이 쿼리를 작성할 수 있습니다.

sql
SELECT * FROM user_reservation_view WHERE rent_user_id = 14 AND id = 13;

장점: 뷰를 사용하면 복잡한 쿼리를 매번 작성할 필요 없이 간단한 쿼리로도 복잡한 데이터를 조회할 수 있습니다.

2. 단순화된 쿼리

  • 복잡한 조인 없이, 필요한 정보를 최소한으로 가져오는 방법도 있습니다. 예를 들어, 단순히 예약 정보와 사용자 정보를 조회하려면 다음과 같이 할 수 있습니다.
sql
SELECT 
    br.id, 
    br.start_date, 
    br.end_date, 
    br.progress, 
    bu.nickname, 
    bc.product_name
FROM 
    bc_reservation br
INNER JOIN 
    bc_rent_user bru ON br.rent_user_id = bru.id
INNER JOIN 
    bc_user bu ON bru.user_id = bu.id
INNER JOIN 
    bc_campingcar bc ON br.product_id = bc.id
WHERE 
    br.rent_user_id = 14 AND br.id = 13;

장점: 최소한의 정보만 가져오므로 쿼리가 더 간단해지고, 이해하기 쉬워집니다.

3. ORM (Object-Relational Mapping) 사용

  • SQL 쿼리를 직접 작성하지 않고, ORM을 사용하여 데이터베이스 작업을 객체 지향적으로 처리할 수 있습니다. 예를 들어, Java에서는 JPA, Python에서는 SQLAlchemy 같은 ORM 라이브러리를 사용하여 복잡한 SQL을 코드로 대체할 수 있습니다.
java
Reservation reservation = reservationRepository.findByRentUserIdAndId(14L, 13L);

장점: ORM을 사용하면 SQL 작성이 부담스러운 경우, 객체 지향적인 방식으로 데이터를 다룰 수 있습니다.

결론:

  • 단순하게 접근하기: 필요한 정보만 선택하여 쿼리를 간소화하거나, 복잡한 쿼리를 뷰로 처리하면 더 쉽게 접근할 수 있습니다.
  • ORM 사용하기: SQL이 너무 어렵게 느껴진다면, ORM과 같은 도구를 통해 더 쉽게 데이터베이스를 다룰 수 있습니다.
